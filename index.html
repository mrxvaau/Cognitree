How it works

The Cognitree Pipeline

Five stages that run on every prompt — turning a forgetful LLM into a precise, state-aware coding agent.

01
🧠
LLM Completes a Task
Claude Code, Qwen Code, or Gemini CLI produces output — code, a decision, a fix, an explanation. This output is the raw material Cognitree works from.
02
⚙️
Memory Extractor Parses Output
Cognitree scans the output and tags extracted data for the correct memory layer — decisions, mistakes, progress, state changes — each to its layer.
extract({ goal: "...", // active objective progress: [...], // what got done mistakes: [...], // what failed — permanent decisions: [...], // why X over Y warnings: [...], // be careful about prompt_state: {} // what AI believes right now })
03
🗄️
Checkpoint Written to Store
An immutable checkpoint is created — memory deltas, code deltas (diffs only), parent pointer, branch ID, confidence score. Full snapshot every 10 checkpoints. Deltas in between.
04
🎯
Context Injector Fires Before Next Prompt
Only relevant layers retrieved and injected. The AI reads a structured block with everything it needs — no re-derivation, no wasted tokens.
[COGNITREE — VoidRoom · Checkpoint 14 · Branch: trunk] GOAL: Implement self-destructing messages PROGRESS: Auth done, schema defined, UI scaffolded NEXT: Add TTL index to messages collection MISTAKES: mongoose.findByIdAndDelete() — hallucinated use findOneAndDelete() instead WARNINGS: TTL index needs background process, not cron DECISION: JWT over sessions — stateless requirement [/COGNITREE]
05
🚀
AI Works with Perfect Precision
The model knows exactly what was built, what failed, what was decided, and what to do next. Zero tokens wasted. Zero repeated mistakes. Zero hallucinations of known-bad paths.
Architecture

Nine Memory Layers

Everything a developer holds in their head while coding — now structured, persistent, and injected with precision before every prompt.

Layer 01 — Active
Goal
What we are building right now. The active objective for this session.
Current feature or fix in progress
Success criteria for this task
Layer 02 — Done
Progress
What is already completed. The AI never re-does finished work.
Completed features and merged logic
Files and modules already implemented
Layer 03 — Permanent
Mistakes
What went wrong. Never cleared. Never forgotten. Never repeated.
Hallucinated APIs and functions
Logic errors that caused bugs
Layer 04 — Blocked
Blockers
What is stuck or unresolved. Needs attention before moving forward.
Unsolved bugs or missing dependencies
Decisions waiting for input
Layer 05 — Immediate
Next
The single immediate next step. One action at a time, always clear.
Exact next function or file to work on
Updated after every checkpoint
Layer 06 — History
Changed
What was updated or refactored and when. Full change log across sessions.
Refactored files with checkpoint reference
API changes, renames, and restructures
Layer 07 — Rationale
Decisions
Why X was chosen over Y. Architecture choices with full reasoning preserved.
Tech stack choices and trade-offs
Pattern choices with full reasoning
Layer 08 — Caution
Warnings
Fragile areas and edge cases. Forwarded into every new branch automatically.
Sensitive files or race conditions
Known gotchas in the stack
Layer 09 — Live
Prompt_State
The AI's active cognitive state. What it currently believes and assumes. Reset per checkpoint.
Active file and function in scope
Current approach and live assumptions
Versioned Cognition

Checkpoint & Branch System

Every prompt is a commit. Every mistake creates a branch. The AI never knows it failed — but the system remembers everything and forwards the warning.

// CHECKPOINT TREE — EXAMPLE
● 0 (BASE) → 1 → 2 → 3 → 4 → 5 → 6 → 7 ← trunk
                                              └─ 8a → 9a → 10a ← failed, branched from 7
                                                                 └─ 8b → 9b → 10b → 11b ● ← active
// WHAT THE AI SEES AT 11b
✓ visible:  0 → 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8b → 9b → 10b → 11b
✗ hidden:   8a, 9a, 10a  (sibling branch — causally isolated)
⚠ injected:  warnings from 10a forwarded into 8b at branch creation
Checkpoint Structure
id:           "cp_11b_a3f9"
parent_id:  "cp_10b_..."
branch_id:  "branch-b"
is_base:    false // full snap every 10
confidence: 0.87 // branch health 0-1
memory_delta: {...} // changed layers only
code_delta:  [...] // file diffs only
Delta Storage
● Full snapshot every 10 checkpoints
● Diffs only between base snapshots
● Unchanged files referenced, never copied
● Reconstruct = walk back to nearest base
● Dead branches kept — teach future sessions
● ~95% storage reduction vs full snapshots
Compatibility

Works With Any Agent

Same memory engine, different injection adapters per tool. No lock-in. Use what you already have.

Claude Code
CLAUDE.md + MCP Server
Cognitree auto-writes memory into CLAUDE.md before each session. Also exposes all 9 memory layers as MCP tools Claude can read and write natively.
Qwen Code
Native Integration · Free
Open source — Cognitree integrates directly into Qwen Code's prompt pipeline. 1000 free API requests per day. No cost to start. Primary integration target.
Gemini CLI
GEMINI.md + State API
Open source — direct integration into prompt construction. GEMINI.md memory file plus Cognitree local state API. Bring your own Gemini API key.
